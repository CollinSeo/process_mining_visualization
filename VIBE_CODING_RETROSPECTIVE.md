# 바이브 코딩으로 프로세스 마이닝 시각화 도구를 만들어보며 느낀 점

## 1. 왜 이 프로젝트를 시작했는가

이번 프로젝트는 프로세스 마이닝(Process Mining) 데이터를 더 직관적으로 이해하기 위한 시각화 도구를 만들어보자는 목적에서 시작했다. 일반적인 이벤트 로그 CSV를 입력으로 받아서, 활동(Activity) 간의 직접적 선후 관계를 보여주는 DFG(Directly-Follows Graph)를 그리고, 여기에 토큰 리플레이 애니메이션까지 붙여서 프로세스의 흐름을 시각적으로 따라갈 수 있게 하는 것이 목표였다.

처음에는 단순히 Python 스크립트로 그래프를 한 번 생성하는 정도를 생각했지만, 진행하다 보니 점점 요구사항이 늘어났다. 정적 PNG뿐 아니라 인터랙티브 HTML이 필요했고, 다시 그것을 웹 GUI 형태로 올려서 CSV 업로드, 컬럼 선택, 필터 조정, 리플레이 속도 조절까지 가능한 형태로 확장되었다. 결과적으로는 단순한 스크립트가 아니라, 작은 로컬 웹 애플리케이션 수준의 결과물이 만들어졌다.

## 2. 프로젝트 배경: 프로세스 마이닝과 시각화

프로세스 마이닝은 정보시스템에 남아 있는 이벤트 로그를 바탕으로 실제 업무 프로세스가 어떻게 흘러가는지를 분석하는 접근이다. 예를 들어 어떤 케이스(case)가 어떤 활동(activity)들을 어떤 순서로 거쳐 갔는지, 특정 경로가 얼마나 자주 발생하는지, 예외 경로는 어떤 것인지 등을 확인할 수 있다.

이번 프로젝트에서 핵심적으로 다룬 개념은 다음과 같다.

### 이벤트 로그(Event Log)

- 한 행(row)이 하나의 이벤트를 의미한다.
- 보통 `case id`, `activity`, `timestamp` 세 가지 핵심 컬럼이 필요하다.
- 같은 case id를 가진 이벤트들을 시간순으로 정렬하면 하나의 프로세스 실행 경로(trace)가 된다.

### DFG(Directly-Follows Graph)

- 어떤 활동 A 다음에 활동 B가 직접 나타났다면 `A -> B`라는 엣지를 만든다.
- 노드는 활동(activity)이고, 엣지는 직접 이어지는 관계를 나타낸다.
- 노드 크기나 색상은 활동 빈도에 따라, 엣지 굵기는 전이 빈도에 따라 조정할 수 있다.

### 토큰 리플레이(Token Replay) 애니메이션

- 각 케이스를 하나의 토큰처럼 보고, 시간의 흐름에 따라 현재 어느 활동에 위치하는지 시각적으로 보여주는 방식이다.
- 이번 프로젝트에서는 단순한 step-by-step 재생이 아니라, 실제 timestamp 기준의 절대 시간 축을 따라 여러 케이스 토큰이 동시에 움직이도록 구현했다.
- 최근에는 이 토큰이 노드 간 직선으로 움직이는 것이 아니라, 화면에 표시된 엣지를 따라 곡선 경로로 이동하도록 개선했다.

이런 시각화는 정적인 표나 빈도 집계보다 훨씬 직관적이다. 특히 프로세스가 복잡해질수록 “어떤 경로가 주요 흐름인지”, “예외 경로는 어디인지”, “시간축 위에서 동시에 어떤 케이스들이 움직이는지”를 눈으로 파악하기 쉬워진다.

## 3. 이번 프로젝트의 최종 형태

현재 프로젝트는 다음 두 가지 실행 방식을 가진다.

### 1) CLI 방식

`dfg_visualizer.py`를 직접 실행해서 CSV 기반으로 PNG/HTML 시각화를 생성한다.

### 2) 웹 GUI 방식

`web_app.py`를 `Streamlit`으로 실행해서,

- CSV 업로드
- 컬럼 선택
- 필터 설정
- PNG/HTML 미리보기
- 결과 다운로드

가 가능한 형태로 사용할 수 있다.

즉, 처음에는 “스크립트”였지만 지금은 “작은 시각화 애플리케이션”에 가까운 형태로 확장되었다.

## 4. 코드 구조와 아키텍처 요약

현재 구조는 아주 복잡한 패키지 아키텍처는 아니지만, 역할은 비교적 분명하다.

### 핵심 파일

- `dfg_visualizer.py`
  - 이벤트 로그 전처리
  - DFG 생성
  - PNG 렌더링
  - 인터랙티브 HTML 생성
  - HTML 후처리 및 JavaScript 삽입
- `web_app.py`
  - Streamlit 기반 웹 UI
  - CSV 업로드, 컬럼 매핑, 옵션 조정, 결과 미리보기/다운로드
- `PROJECT_CONTEXT.md`
  - 프로젝트 컨텍스트와 변경 이력, 다음 세션 인수인계용 문서

### 내부 흐름

대략적인 처리 흐름은 다음과 같다.

1. CSV 이벤트 로그를 읽는다.
2. `case id`, `activity`, `timestamp` 컬럼을 기준으로 정렬 및 정제한다.
3. 각 케이스의 trace를 기준으로 directly-follows 관계를 계산한다.
4. 노드 빈도, 엣지 빈도, 전이 확률을 계산한다.
5. PNG용 DFG를 `matplotlib + networkx`로 그린다.
6. HTML용 DFG를 `pyvis`로 생성한다.
7. HTML 결과물에 후처리 JavaScript를 삽입해 다음 기능을 붙인다.
   - 복잡도 조정
   - 노드/엣지 시각 크기 조정
   - 토큰 리플레이
   - 대규모 로그 최적화 모드
   - 토글형 컨트롤 패널

### 현재 아키텍처의 특징

장점:

- 아이디어를 빠르게 실험하고 결과를 바로 확인하기 좋다.
- 하나의 핵심 파일에서 전체 흐름을 추적할 수 있다.
- 시각화와 인터랙션을 빠르게 붙이기 좋은 구조다.

한계:

- `dfg_visualizer.py` 안에 로직이 많이 모여 있다.
- 데이터 처리, 렌더링, HTML 생성, JS 삽입이 한 파일에 함께 있어 규모가 더 커지면 유지보수가 어려워질 수 있다.

즉, 빠르게 시도하고 결과를 만드는 데는 매우 적합했지만, 장기적으로는 모듈 분리와 명확한 구조화가 필요해 보인다.

## 5. 바이브 코딩으로 만들어가며 느낀 신기함

이번 프로젝트에서 가장 인상적이었던 점은, 내가 머릿속으로 생각한 기능들을 대화형으로 계속 추가해 가면서 실제 프로그램으로 빠르게 구체화할 수 있었다는 점이다.

예를 들면 다음과 같은 흐름이 가능했다.

- 단순 DFG 생성 스크립트 만들기
- 인터랙티브 HTML 추가하기
- 복잡도 슬라이더 넣기
- 리플레이 애니메이션 넣기
- 리플레이를 절대 시간 기반으로 변경하기
- 대규모 로그에서 느려지는 문제 최적화하기
- Streamlit GUI로 감싸기
- 노드 크기/폰트/엣지 굵기 슬라이더 추가하기
- 토큰이 엣지를 따라 움직이도록 바꾸기

이런 요구사항 변화가 전통적인 방식이었다면 설계, 구현, 수정, 재테스트에 시간이 꽤 걸렸을 텐데, 바이브 코딩에서는 “원하는 방향을 말하고 바로 구현/검증하는” 리듬이 형성되었다. 특히 시각적인 결과물을 다루는 프로젝트에서는 이 피드백 루프가 굉장히 강력했다.

무엇보다 신기했던 점은, 내가 처음부터 모든 세부 설계를 명확히 갖고 있지 않아도, 요구를 조금씩 구체화하면서 결과물을 발전시킬 수 있었다는 점이다. 생각이 정리되는 속도와 구현이 진행되는 속도가 비슷하게 맞물리는 느낌이 있었다.

## 6. 어려웠던 점과 한계

물론 좋은 점만 있었던 것은 아니다. 실제로 진행하면서 몇 가지 어려움도 분명히 느꼈다.

### 1) 세션이 끊기면 컨텍스트가 이어지지 않는 문제

가장 크게 느낀 어려움은 세션이 바뀌거나 다시 시작했을 때, 이전 대화에서 쌓인 맥락이 자연스럽게 이어지지 않는다는 점이었다.

예를 들어 이전 세션에서는 이미

- 왜 이런 구조로 만들었는지
- 어떤 파일이 핵심인지
- 어떤 최적화를 했는지
- 어떤 문제를 해결하다가 현재 상태에 도달했는지

를 다 알고 있었는데, 새 세션에서는 다시 설명하지 않으면 이 맥락이 사라진다. 그러면 같은 내용을 다시 파악하거나, 이미 결정된 방향과 다른 쪽으로 해석될 가능성이 생긴다.

바이브 코딩의 장점이 빠른 반복이라면, 이런 컨텍스트 손실은 그 흐름을 끊는 요소였다.

### 2) 빠른 구현과 구조적 품질 사이의 균형

바이브 코딩은 빠르지만, 그만큼 코드 구조가 자연스럽게 누적되는 경향이 있다. 요구사항이 추가될수록 “지금 당장 작동하게 만드는 것”은 쉬운데, 장기적으로 봤을 때 얼마나 잘 정리된 구조인지에 대해서는 별도 관리가 필요하다.

이번 프로젝트도 실제로는 꽤 잘 동작하지만, 시간이 지나 요구사항이 더 붙으면 `dfg_visualizer.py`를 모듈 단위로 나누는 작업이 필요할 것으로 보인다.

### 3) 성능 이슈는 결국 직접 부딪혀야 알 수 있다는 점

처음에는 HTML 리플레이도 금방 될 줄 알았지만, 실제로는 로그가 커지면서 성능 문제가 크게 드러났다.

- replay payload가 너무 커서 HTML 로딩이 느려짐
- 토큰이 많아지면서 애니메이션이 무거워짐
- 필터를 움직일 때 재계산이 많아짐

이런 문제는 처음 구현 단계에서는 잘 보이지 않았고, 실제 데이터를 붙이고 반복해서 수정하는 과정에서 드러났다. 결국 성능은 “일단 만들어 보고, 병목을 확인하고, 다시 구조를 바꾸는 과정”이 필요했다.

## 7. `PROJECT_CONTEXT.md`로 보완했던 점

이번 프로젝트에서 꽤 유용했던 장치는 `PROJECT_CONTEXT.md` 파일이었다.

이 파일은 단순 README가 아니라, 사실상 다음 세션을 위한 인수인계 문서 역할을 했다.

여기에 정리한 내용은 예를 들면 다음과 같다.

- 프로젝트 목적
- 주요 파일과 역할
- 핵심 함수 설명
- CLI/GUI 실행 방식
- 지금까지 적용된 주요 변경 사항
- 성능 최적화 내역
- 리플레이 구조 변경 내용
- 다음 세션에서 바로 이어가기 위한 맥락

이 문서를 통해 얻은 장점은 명확했다.

### 1) 세션이 바뀌어도 빠르게 재진입 가능

새 세션에서 처음부터 다시 코드베이스를 모두 해석하지 않고, 우선 `PROJECT_CONTEXT.md`를 읽으면 전체 방향과 상태를 빠르게 이해할 수 있었다.

### 2) “왜 이렇게 만들었는지”가 남는다

코드만 보면 결과는 보여도 의도는 잘 안 보일 때가 많다. 특히 바이브 코딩 과정에서는 변경 속도가 빨라서, 나중에 보면 “왜 이 최적화를 넣었지?”, “왜 이 UI가 생겼지?” 같은 질문이 생긴다. 이런 부분을 문서가 보완해 줬다.

### 3) 컨텍스트 손실을 줄여준다

대화형 개발에서 가장 큰 문제 중 하나가 맥락 손실이라면, 이런 문서는 그 손실을 꽤 줄여주는 안전장치 역할을 한다.

이번 경험을 통해, 바이브 코딩에서는 코드 자체만큼이나 “컨텍스트 문서화”가 중요하다는 점을 강하게 느꼈다.

## 8. 이번 경험에서 얻은 개인적인 인사이트

이번 프로젝트를 하면서 느낀 점을 한 문장으로 요약하면,

“바이브 코딩은 생각보다 훨씬 빠르게 결과물을 만들 수 있지만, 컨텍스트 관리와 구조화 없이는 점점 불안정해질 수 있다.”

였다.

좋았던 점은 다음과 같다.

- 아이디어를 빠르게 결과물로 옮길 수 있다.
- 구현과 피드백의 루프가 짧다.
- 시각화나 도구형 프로젝트에서 특히 강력하다.
- 처음 해보는 형태의 프로그램도 훨씬 낮은 진입장벽으로 시도해볼 수 있다.

아쉬웠던 점은 다음과 같다.

- 세션 단절 시 컨텍스트 유지가 어렵다.
- 구조적 설계보다 국소 최적화가 앞설 수 있다.
- 프로젝트가 커질수록 문서화와 정리가 필수적이다.

결국 바이브 코딩은 “아무 생각 없이 막 만드는 방식”이라기보다는, 오히려 더 자주 맥락을 정리하고 방향을 보정해야 잘 쓸 수 있는 방식이라는 생각이 들었다.

## 9. 앞으로 해보고 싶은 방향

이번에는 비교적 탐색적으로, 요구사항을 하나씩 붙여가며 발전시키는 형태로 프로젝트를 진행했다. 이 방식 자체가 매우 유용했지만, 다음 단계에서는 조금 더 체계적인 접근을 해보고 싶다.

특히 장기적으로는 다음과 같은 방향을 생각하고 있다.

### 1) 프로젝트 시작 단계의 구조 설계를 더 잘하기

처음부터

- 데이터 처리 계층
- 시각화 계층
- 웹 UI 계층
- 성능 전략
- 문서화 전략

를 조금 더 의식적으로 나누고 시작하면, 중간에 구조가 커져도 덜 흔들릴 것 같다.

### 2) 하네스 엔지니어링(Harness Engineering) 같은 방법론 도입

앞으로는 단순히 기능을 추가하는 것에서 끝나지 않고,

- 재현 가능한 실행 경로
- 검증 가능한 테스트 입력
- 문맥을 유지하는 문서화
- 반복 가능한 작업 단위

를 갖춘 방식으로 프로젝트를 진행해보고 싶다.

즉, 바이브 코딩의 속도는 유지하되, 그 위에 하네스 엔지니어링 같은 방법론을 얹어서 처음부터 더 체계적으로 설계하고 결과물을 만드는 방식을 시도해보고 싶다.

이렇게 하면 단발성 실험이 아니라, 반복 가능하고 확장 가능한 개발 흐름으로 발전시킬 수 있을 것 같다.

## 10. 마무리

이번 프로젝트는 “바이브 코딩으로 정말 어디까지 갈 수 있을까?”를 직접 체감해본 경험이었다.

결론적으로는,

- 꽤 복잡한 시각화 도구도 빠르게 만들 수 있었고
- 중간에 요구사항이 많이 바뀌어도 계속 따라갈 수 있었으며
- 성능 최적화, UI 개선, 웹 앱 전환까지 실제로 해낼 수 있었다.

동시에,

- 컨텍스트 유지의 중요성
- 문서화의 필요성
- 장기 구조 설계의 중요성

도 분명하게 느낄 수 있었다.

사내 바이브 코딩 연구회 차원에서 보더라도, 이 경험은 “바이브 코딩이 정말 생산적인가?”라는 질문에 대해 꽤 긍정적인 사례가 될 수 있다고 생각한다. 다만 그 생산성을 지속 가능하게 만들려면, `PROJECT_CONTEXT.md` 같은 컨텍스트 문서화와 더 나아가 하네스 엔지니어링 같은 체계적 방법론이 함께 필요하다는 점도 같이 공유하고 싶다.

이번 프로젝트는 그 출발점으로 꽤 의미 있는 사례였다.
